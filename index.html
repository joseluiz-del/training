        You create a yard allocation by defining a set of criteria that match containers you want. This is called an allocation group.       2. Planning containers to the yard

        In yard planning, the goal is to have XPS assign containers to specific yard positions automatically using yard allocations. Of course, you can also plan a container manually by selecting it and clicking a slot in a yard view. You can use the Pointer or Flow tools available in the XPS toolbar to position containers one by one or in groups. Use manual planning if XPS cannot find an allocation and one or more containers fail to deck.


        To plan a container automatically, you first create the allocations that XPS will use to match containers to yard space. Typically, XPS automatically assigns a position to a container received at the gate upon receiving the information from the host computer. XPS also automatically assigns positions when you select containers and choose Planning

        Assign Position. You do this for a vessel discharge or moves within the yard.

        XPS assigns container positions according to group priority. This means that if a container matches more than one allocation group, XPS assigns it to the first matching group and range on the list in the Yard Allocations window.


        To automate yard planning and control even further, you can license the Navis Expert Decking module. Because Expert Decking assigns an optimum position for containers in real time, you achieve breakthroughs in efficient use of yard space and CHE. Expert Decking expands the XPS allocation model by adding stacking and section factors and pena/ty strategies to the existing allocation process.


        For containers that are located in shared yard blocks that are not owned by the current facility, XPS displays a dimmed container icon in the graphical yard views and does not allow you to plan these containers. If debug mode is on, XPS also displays section factor and stacking factor attributes on containers owned by another facility. This helps admin users to understand and debug Expert Decking re-handle decisions.


      3. Managing and retaining container plans

        It is possible to differentiate yard groups in XPS. This allows you to manage and retain container plans if the container does not complete its move to the intended "Sub Group". The plan is retained for both inventory based updates as well as EC/Automation based updates. Only position updates that are manually completed are marked as complete. This logic allows you to make differentiations for wheels and for ground in the yard source file to accommodate the retention of the plan.

        For grounded slots, specify the value G%%% and for wheeled slots, specify W%%% in the code characters within the block definition of the yard source file. Leave blank if your operation does not need to differentiate wheeled and grounded containers. Modify the SPARCS Settings.txt file to add the setting RSTRFT which means "Restrict move completion from-to yard code e.g., WG= no wheel-to-ground, GW= no ground-to-wheel. The default value is Blank meaning "restrict all wheel vs ground moves". Set the value to WG to enforce wheel to ground move. That is, if a move was destined to go to a grounded position and was placed in a different subgroup area, e.g. wheels, the plan will be retained and only one type of move can be restricted. When left as blank value, all moves will retain the plan when the move has been completed to a subgroup area different from where it was planned to.


        Depending on how the "code chars" have been defined, the following represents the expected behavior:


        • Container inventoried in wheel position with planned position to ground...should keep plan position

        • Container inventoried in ground position with planned position to wheel. should keep plan position

        • Container inventoried in wheel position with planned position to wheel position. should lose the plan position

        • Container inventoried in ground position with planned position to ground position. should lose the plan position

Containers planned to a position display on ordered lists of work instructions called u/ork queues. XPS places a container in a yard, road, rail, or vessel queue depending on the type of move.


In a trouble-free yard plan, all containers fit at least one allocation range and there is enough space for each container. But if a container does not match any group or range, or there is not enough space in the matching ranges, XPS adds it to the Fail to Deck logical block. The XPS Notices window displays a reason for the failure.


You use XPS diagnostic features to correct problems as they occur and keep yard operations flowing at full speed.
